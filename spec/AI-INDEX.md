# AI-INDEX: как работать со спецификацией при создании сайта Eagle Security

Этот файл — инструкция для AI (и подсказка для человека): в каком порядке читать документы и как по ним собирать сайт.

---

## Структура репозитория

```
apps/site-eaglesecurity/
├── spec/           # Спецификация и ТЗ (не трогать при сборке, только читать)
│   ├── 00-site-overview.md   # Обзор сайта, закон, стиль, i18n, deploy
│   ├── 01-tz-template.md     # Шаблон ТЗ (заполняется заказчиком)
│   ├── AI-INDEX.md           # Этот файл
│   └── README.md             # Оглавление spec
└── src/            # Исходники сайта (здесь генерировать код и контент)
```

---

## Порядок работы при создании сайта по ТЗ

1. **Прочитать обзор**  
   [00-site-overview.md](00-site-overview.md) — даёт общие правила: обязательные разделы (Impressum, Datenschutz), стиль, два языка, деплой/серв. Не задаёт конкретный набор страниц.

2. **Прочитать заполненное ТЗ**  
   [01-tz-template.md](01-tz-template.md) — после заполнения заказчиком здесь будут: домен, страницы, контент, цвета, шрифты, технология, скрипты. Все решения по структуре и дизайну брать из ТЗ.

3. **Разрешать конфликты**  
   Если в ТЗ что-то не указано или отмечено «на усмотрение»:
   - по закону (Impressum, Datenschutz) — следовать [00-site-overview.md](00-site-overview.md);
   - по дизайну и наборам страниц — либо спросить, либо выбрать разумный вариант и зафиксировать его в коде/комментарии.

4. **Реализовывать в `src/`**  
   - Структура `src/` — как у референсного сайта Int64 (или по ТЗ): контент (Markdown/данные), шаблоны, стили, скрипты `serve.sh` / `deploy.sh`.
   - Обязательно: главная, Impressum, Datenschutz; остальные страницы — по списку из ТЗ.
   - Два языка: отдельные файлы или данные по локалям (DE/EN), переключатель в футере (или где указано в ТЗ).
   - Стиль: цвета и шрифты из ТЗ; при отсутствии — минимализм, mono, тёмная тема по обзору.

5. **Деплой и локальный просмотр**  
   - В `src/` должны быть скрипты (или команды в README): один — для localhost, один — для деплоя.
   - PathPrefix для GitHub Pages (если сайт в подпапке) — из ТЗ или из настроек репозитория.

6. **После реализации**  
   - Не менять заполненное ТЗ в `spec/` без запроса заказчика.
   - Изменения вносить только в `src/` и при необходимости в README/документацию в `src/`.

## Референс: сайт Int64

Для реализации можно ориентироваться на структуру и подход из репозитория **Int64** (визитка):

- Статика на Eleventy, контент в Markdown, данные в `_data/`, локали DE/EN.
- В футере: ссылки на разделы + переключатель языка.
- `serve.sh` — локальный просмотр, `deploy.sh` — публикация в `gh-pages` с автоопределением pathPrefix для GitHub Pages.
- Стиль: моношрифт, тёмный фон, зелёный текст; логотип ASCII, консольный псевдо-UI.

Сайт Eagle Security по ТЗ может отличаться набором страниц и акцентами дизайна, но требования по закону, i18n и деплою из обзора остаются базой.
